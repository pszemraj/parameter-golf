- Commit: `de95b50`
- Phase: trigram-memory runtime allocation cleanup
- Main observations:

## 2026-04-26T04:24:00-04:00

- Commit: `f528444`
- Phase: exact parallel trigram-memory counting option
- Main observations:
  - The dense trigram-memory builder was serial over files/chunks; full K2/K4
    counts spent most of their time in one Python process running
    `np.unique(..., return_counts=True)`.
  - Added `TRIGRAM_COUNT_WORKERS` / `--count-workers` for exact worker-local
    dense count tables over contiguous shard blocks.
  - This avoids unsafe shared-table writes. Workers count privately, then the
    parent reduces the dense tables.
  - The mode is memory/disk hungry: at SP1024, each worker needs about
    `4.29 GB` for its local count table before reduction.
  - Default remains `1` worker so running jobs and low-memory launches are not
    silently changed.
