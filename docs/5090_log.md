## 2026-04-26T03:36:00-04:00

- Commit: `5340173`
- Phase: trigram memory table-cache backfill
- Main observations:
  - If a matching output memory spec already exists but the reusable counted
    table cache is missing, the builder now backfills the table cache from the
    existing `spec.pt` before returning.
  - This prevents compatible geometry shapes from repeatedly doing the full CPU
    trigram count when one spec was already built but the shared table cache was
    absent or deleted.
- Validation:
  - capped `tools/build_trigram_memory_spec.py` smoke at `--max-tokens 200000`
    confirmed existing-spec table-cache backfill after deleting the cached
    table file.
- Transformer-regression note:

  - This is cache behavior only.

## 2026-04-26T03:20:31-04:00

- Commit: `1f51662`
- Phase: final-week protocol naming and cache-safety cleanup
- Main observations:
  - Trigram tables are named as spec-resident memory, not as an external add-on.
  - The final `spec.pt` contains the trigram memory tensors; the separate table
    cache is only a reusable counted-tensor build artifact for compatible specs.
  - The memory spec cache and counted-table cache include a training-shard
    fingerprint so partial or changed datasets do not silently reuse stale
    statistics.
  - Serious launchers print both the memory-spec cache root and the counted
    table cache root so a full-data recount is distinguishable from training.
  - Legacy `trigram_sidecar` config keys are accepted only for old-run
    analysis; new configs and launchers write `trigram_memory`.
- Follow-up docs stamp:
  - Commit `7285ee6` recorded validation.
- Transformer-regression note:
  - None.
  - This is terminology, cache protocol, and launcher clarity only.
